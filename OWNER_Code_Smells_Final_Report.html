Phase 1: Repository Understanding

Project Overview

OWNER is a Java library designed to reduce boilerplate in application configuration. Instead of manually reading Java properties files, users define an interface extending Config. OWNER creates a dynamic proxy that maps interface methods to configuration values. The main public entry point is ConfigFactory.create(MyConfig.class).

The project is a multi-module Maven repository. The core module contains the main runtime behavior. Additional modules provide optional integrations, Java 8 support, examples, documentation, and distribution assembly.

Architecture Overview

OWNER follows a dynamic-proxy based architecture. A user defines a configuration interface, and OWNER creates an implementation at runtime. Method calls are intercepted, mapped to property keys, resolved from loaded property sources, optionally preprocessed or decrypted, and converted from strings into the requested return type.

Execution Flow

  1. User defines an interface extending Config.
  2. User calls ConfigFactory.create(MyConfig.class).
  3. DefaultFactory constructs PropertiesManager, optional JMXSupport, and PropertiesInvocationHandler.
  4. A Java dynamic proxy is created.
  5. Each method call is intercepted by PropertiesInvocationHandler.invoke.
  6. Delegate methods are routed first. Normal config methods resolve a property key.
  7. The value is loaded from PropertiesManager, processed, decrypted if needed, formatted, and converted.
  8. The converted value is returned to the caller.

Dependency Overview

The core module has a small runtime footprint. Its direct dependencies in owner/pom.xml are test-scoped dependencies: JUnit, Hamcrest, Mockito, and commons-codec. Optional modules add dependencies such as Apache Curator for ZooKeeper support. The parent build uses Maven Compiler, Surefire, Javadoc, Source, Bundle, Assembly, and reporting plugins.

Change 2: Remove Duplicate Load Workflow

File: owner/src/main/java/org/aeonbits/owner/PropertiesManager.java

Lines after refactoring: 505-543

Before Code:

public void load(InputStream inStream) throws IOException {
    writeLock.lock();
    try {
        Properties loaded = new Properties();
        loaded.load(inStream);
        performLoad(keys(loaded), loaded);
    } catch (RollbackBatchException ex) {
        ignore();
    } finally {
        writeLock.unlock();
    }
}

After Code:

public void load(InputStream inStream) throws IOException {
    load(new PropertiesLoader() {
        public void load(Properties props) throws IOException {
            props.load(inStream);
        }
    });
}

private void load(PropertiesLoader loader) throws IOException {
    writeLock.lock();
    try {
        Properties loaded = new Properties();
        loader.load(loaded);
        performLoad(keys(loaded), loaded);
    } catch (RollbackBatchException ex) {
        ignore();
    } finally {
        writeLock.unlock();
    }
}

Benefits: The lock and event workflow now exists in one place. This reduces future maintenance risk and keeps behavior consistent between stream and reader loading.

Change 3: Simplify Property Resolution Pipeline

File: owner/src/main/java/org/aeonbits/owner/PropertiesInvocationHandler.java

Lines after refactoring: 77-104

Before Code:

String key = expandKey(method, args);
String value = propertiesManager.getProperty(key);
if (value == null && !isFeatureDisabled(method, VARIABLE_EXPANSION)) {
    String unexpandedKey = key(method);
    value = propertiesManager.getProperty(unexpandedKey);
}
...
Object result = convert(method, method.getReturnType(),
        format(method, propertiesManager
            .decryptIfNecessary(method, expandVariables(method, value)),
            args));

After Code:

private Object resolveProperty(Method method, Object... args) {
    String value = getProperty(method, args);
    if (value == null)
        return null;
    Object result = convertValue(method, prepareValue(method, value, args));
    if (result == NULL) return null;
    return result;
}

Benefits: The method now expresses the high-level flow clearly. The extracted methods preserve the exact lookup, preprocessing, expansion, decryption, formatting, and conversion order.

## Summary

This PR improves maintainability in OWNER's core runtime path by applying small,

- PropertiesManager constructor mixed several initialization responsibilities.

- Extracted primitive conversion into convertPrimitive.
- Updated compiler release to Java 8 for compatibility with modern JDKs.
- Reworked a fragile Mockito Properties spy test into a deterministic test helper.

- Reduced duplication.
- Improved readability of core configuration loading and invocation flow.
- Preserved public API and runtime behavior.
- Restored successful build/test execution on JDK 21.

Phase 8: Final Academic Report

Repository Overview

OWNER is a mature Java configuration library with a focused purpose: simplifying the use of properties-based configuration. It uses interface-based configuration definitions and runtime proxy generation to provide type-safe access to configuration values.

Architecture Analysis

The architecture is centered on dynamic proxy dispatch. This design enables a concise public API but places significant responsibility on runtime classes such as PropertiesManager, PropertiesInvocationHandler, and Converters. These classes are therefore natural locations for maintainability concerns.

Code Smell Analysis

The most important smells were large class, duplicate code, complex method, broad utility responsibilities, and global mutable state. Not every smell was refactored because several are tied to public API compatibility or intentional legacy design. The implemented changes focused on high-confidence, behavior-preserving improvements.

Refactoring Decisions

The selected refactorings were deliberately conservative. They targeted internal private methods, avoided public API changes, and used the existing style of the codebase. This reduced risk while still providing meaningful maintainability improvements.

Validation Results

Functional validation was successful. The commands ./mvnw clean test and ./mvnw clean install completed successfully after the build compatibility and test compatibility fixes. This indicates that behavior remained stable according to the repository's existing test suite.

Tools Used

  • Manual code review
  • rg for source searching
  • find and wc for repository inventory and file size checks
  • git diff for change review
  • Maven Wrapper for build and test validation
  • JUnit and Mockito test suite
  • LibreOffice for document generation

Challenges Faced

  • The original build target was Java 6, which is not supported by the installed JDK 21 compiler.
  • A Mockito spy on java.util.Properties failed under the current runtime and required a deterministic test helper.
  • Some smells, such as the large public Config interface, were real but not suitable for safe refactoring because of public API compatibility.

Reflection and Lessons Learned

This study shows that valuable refactoring does not always require major redesign. Small, carefully selected internal changes can reduce duplication, clarify execution flow, and improve maintainability while preserving behavior. The existing test suite was essential because it made it possible to validate that the refactoring did not break configuration loading, property conversion, reload behavior, events, or optional modules.

Conclusion

The OWNER repository contains a generally coherent architecture but also several maintainability issues typical of mature libraries: central runtime classes grow large, compatibility concerns preserve older patterns, and utility logic accumulates over time. The implemented refactorings improved the most assignment-relevant issues while maintaining functional correctness. The final build and test results confirm that the repository remains operational after refactoring.
